Конспект лекции по лабе: 
Введение в базы данных нового SQL 
Базы данных нового SQL используют разнообразные модели данных для доступа и управления информацией.
Оптимизированы для приложений с большим объёмом данных, низкой задержкой и гибкими моделями.
Смягчают жёсткие требования к непротиворечивости данных.
Пример моделирования схемы 
В традиционной базе данных запись о книге разделяется на несколько частей и хранится в отдельных таблицах.
Связи между таблицами определяются ограничениями первичных и внешних ключей.
В базе данных нового SQL запись о книге хранится как документ JSON, где все данные объединены в одном документе.
Горизонтальная масштабируемость 
Горизонтальная масштабируемость подразумевает увеличение количества вычислительных машин или виртуальных машин.
Вертикальная масштабируемость предполагает увеличение памяти или мощности процесса на имеющейся ноде.
Преимущества баз данных нового SQL 
Гибкость: быстрые схемы и поэтапная реализация.
Масштабируемость: использование распределённых кластеров.
Высокая производительность: оптимизация для конкретных моделей данных и шаблонов доступа.
Широкие функциональные возможности: специальные API и типы данных.
Свойства BASE 
Базовая доступность: каждый запрос гарантированно завершается успешно или безуспешно.
Гибкое состояние: состояние системы может изменяться со временем без ввода новых данных.
Согласованность в конечном счёте: данные могут быть рассогласованы, но приходят к согласованию через некоторое время.
Теорема CAP и её применение 
Теорема CAP: в распределённых вычислениях можно обеспечить только два из трёх свойств: согласованность, доступность или устойчивость к разделению.
Системы на основе BASE не подходят для всех приложений, например, для биржевых и банковских систем требуются транзакции.
Проектировщики NoSQL систем жертвуют согласованностью данных ради достижения доступности и устойчивости к разделению.
Особенности решений нового SQL 
Настройка характеристик доступности и согласованности для отдельных запросов.
Применение различных типов хранилищ.
Возможность разработки базы данных без задания схем.
Линейная масштабируемость: добавление процессоров увеличивает производительность.
Инновационность: новые возможности для хранения и обработки данных.
Эволюция термина «новый SQL» 
Изначально новый SQL позиционировался как «без SQL», но позже термин изменился на «not-only SQL».
Новый SQL открывает много возможностей для хранения и обработки данных.
Введение в новые SQL-решения 
Описание схемы данных в новом SQL может осуществляться через различные структуры данных: хэш, таблицы, деревья и другие.
Выделяются четыре основных типа систем: ключ-значение, семейство столбцов, документно ориентированные и графовые.
Ключ-значение 
Модель ключ-значение использует ключ для доступа к значению, принцип хранения похож на хэш-таблицу.
Примеры использования: хранение изображений, специализированные файловые системы, кэширование объектов.
Примеры СУБД: Berkeley DB, Memcached, Redis, Amazon Dynamo DB.
Поддерживают высокую разделяемость и горизонтальное масштабирование.
Семейство столбцов 
Данные хранятся в виде разреженной матрицы, строки и столбцы используются как ключи.
Типичное применение: веб-индексирование, задачи с большими данными.
Примеры СУБД: Apache HBase, Apache Cassandra, Scilla DB, Apache Kudu Hyper Table.
Обеспечивают высокую производительность в оперативных сценариях.
Документно ориентированные 
Служат для хранения иерархических структур данных.
Применяются в системах управления содержимым, издательском деле, документальном поиске.
Примеры СУБД: CouchDB, Couchbase, MongoDB, Exist, Berkeley DB XML.
Данные часто представлены как объекты или документы в формате, подобном JSON.
Графовые 
Применяются для задач с большим количеством связей между данными.
Примеры: соцсети, выявление мошенничества.
Примеры СУБД: Neo4G, OrientDB, Allegro Graph, Blaze Graph, Infinite Graph.
Поддерживают ACID и специализированные языки запросов, такие как Gremlin, Cypher, GraphQL.
Поисковые БД 
Используются для визуализации и анализа автоматически генерируемых потоков данных.
Индексируют и агрегируют частично структурированные журналы и метрики.
Применяются в операционном мониторинге, устранении неисправностей, отслеживании стека распределённых приложений и оптимизации затрат.
Redis 
Популярна как система кэширования благодаря быстрому поиску по ключу.
Медленно извлекает сводную статистику по многим данным.
Поддерживает очереди, сравнимо с Kafka и RabbitMQ.
База данных на основе пар ключ-значение 
Хранит данные как совокупность пар ключ-значение, где ключ служит уникальным идентификатором.
Обеспечивает высокую разделяемость и горизонтальное масштабирование.
Отличается от реляционных БД отсутствием определённой структуры данных.
Домены и их особенности 
Домены могут содержать любую структуру данных, записи идентифицируются по ключу.
Каждая запись имеет динамический набор атрибутов.
Между доменами и внутри одного домена отношения явно не определены.
Проблемы дублирования данных 
Данные могут дублироваться, если связанные данные хранятся в одном домене.
Пример: заказ ссылается на клиента, данные о котором хранятся в том же домене.
Особенности хранилищ без референсов 
В таких хранилищах нет понятия референсов и ссылок.
Данные каждого заказа хранятся изолированно.
Клиенты каждого заказа дублируются, даже если это один и тот же клиент.
Преимущества подхода «ключ-значение» 
Дублирование данных между доменами считается приемлемым из-за относительной дешевизны дискового пространства.
Все связанные данные хранятся в одном месте, что улучшает масштабируемость.
Нет необходимости соединять данные из разных таблиц.
Пример хранения данных 
По ключу 1 хранится один набор атрибутов, по ключу 2 — другой.
Связи между сущностями всё равно нужны, даже если они не определены в модели данных.
Запись о заказе содержит ключи, указывающие на соответствующие записи о покупателе и товаре.
Обеспечение целостности данных 
СУБД не может контролировать целостность связей, это ложится на приложение.
Можно удалять покупателей и товары, которые они заказывали.
Бизнес-логика и логика для поддержки целостности данных содержатся в коде приложений.
Доступ к данным и интеграция 
Данные создаются, обновляются, удаляются и запрашиваются через API-методы.
Некоторые реализации предоставляют SQL-подобный синтаксис для фильтрации данных.
Для взаимодействия с приложениями используются SAP или REST API.
Масштабируемость и преимущества 
Хранилища типа «ключ-значение» легко масштабируются и динамически расширяются.
Они подходят для облачных сервисов и многопользовательских веб-платформ.
Пользователи платят только за использованные ресурсы, что позволяет вендорам увеличивать размер платформы в зависимости от нагрузки.
Несовместимость моделей данных 
Реализационная модель данных и объектная модель кода часто строятся по-разному, что приводит к несовместимости.
ORM-фреймворки легко преодолевают эту несовместимость.
Данные можно разложить по реализационным таблицам, которые можно нормализовать до третьей нормальной формы.
Решение проблемы несовместимости 
Разработчики решают проблему несовместимости с помощью ORM-логики, которая отображает реализационную модель в объектную.
Этот процесс может занять значительное время, которое можно было бы потратить на разработку приложения.
Хранилища типа ключ-значение могут уменьшить время разработки, так как данные отображаются в объекты более естественно.
Недостатки хранилищ типа ключ-значение 
В реляционных БД ограничения гарантируют целостность данных на низком уровне.
В хранилищах типа ключ-значение контроль целостности данных полностью лежит на приложениях, что может привести к ошибкам.
Ошибки в коде могут легко вызвать проблемы с целостностью данных в хранилищах типа ключ-значение.
Преимущества реляционных БД 
Реляционные БД вынуждают разработчиков проходить через процесс разработки модели данных, что делает данные независимыми от приложения.
Это позволяет другим приложениям использовать те же данные без изменений в модели базы.
Для достижения аналогичного эффекта в хранилищах типа ключ-значение требуется проектирование классов, основанных на естественной структуре данных.
Совместимость и стандарты 
Облачные хранилища имеют меньше общих стандартов по сравнению с традиционными БД.
Переход между разными поставщиками услуг может быть сложным из-за различий в API и интерфейсах запросов.
Современные хранилища типа ключ-значение находятся в стадии развития, что создаёт дополнительные риски.
Ассоциативный массив 
Ассоциативный массив — это абстрактный тип данных, позволяющий хранить пары ключ-значение и поддерживать операции добавления, поиска и удаления.
Ассоциативный массив не может хранить две пары с одинаковыми ключами.
Основные операции: Insert, Find, Remove.
Расширения ассоциативного массива 
Популярные расширения: Clear, Iterate, Min, Max.
Для операций Min и Max необходимо, чтобы на ключах была определена операция сравнения.
Реализации ассоциативного массива 
Простая реализация основана на обычном массиве, элементами которого являются пары ключ-значение.
Для ускорения поиска можно упорядочить элементы массива по ключу и использовать бинарный поиск.
Наиболее популярные реализации основаны на деревьях поиска, например, на красных чёрных деревьях.
Хэш-таблицы 
Хэш-таблицы имеют среднее время выполнения операций O1, но не гарантируют высокую скорость выполнения отдельных операций.
Время операции Insert в худшем случае оценивается как OATN из-за возможности коллизий хэшей.
Хэш-таблицы не позволяют быстро реализовать дополнительные операции min-max и обход всех хранимых пар в порядке возрастания или убывания ключей.
Введение в Redis 
Redis — это резидентная система управления базами данных класса «новый SQL» с открытым исходным кодом.
Работает со структурами данных типа «ключ-значение».
Используется для баз данных, кэшей и брокеров сообщений.
Реализует механизм обмена сообщениями через PUB/SUB.
Производительность и разработка 
Ориентирован на максимальную производительность атомарных операций.
Обещает до 100 тысяч SET и GET-запросов в секунду на Linux-серверах начального уровня.
Написан на C, интерфейсы доступа созданы для большинства основных языков программирования.
Спонсировался VMware, затем Pivotal и Redis Labs.
Механизмы обмена сообщениями и репликация 
Предоставляет операции для реализации механизма обмена сообщениями в шаблоне «издатель-подписчик».
Поддерживает репликацию данных с основных узлов на несколько подчинённых.
Работает на большинстве POSIX-систем, включая Linux, BSD, MacOS X.
Поддержка языков программирования и типов данных 
В версии 2.6.0 добавлена поддержка языка Lua.
Поддерживаются абстрактные типы данных: строки, списки, множества, хэш-таблицы, упорядоченные множества.
Данные хранятся в виде словаря с ключами и значениями.
Восстановление и репликация данных 
Восстановление данных осуществляется через механизм снимков и журнал упреждающей записи.
Репликация данных помогает защитить данные и увеличить производительность.
Система репликации не поддерживает автоматическую отказоустойчивость, но есть система Redis Sentinel для мониторинга и автоматического переключения узлов.
Типы данных и базовые операции 
Строки: бинарно безопасные, ограничены размером в полгигабайта.
Списки: упорядоченные списки строк.
Множества: неупорядоченные множества строк, поддерживают операции вставки, проверки вхождения элемента, пересечения и разницы множества.
Хэш-таблицы: ассоциативные массивы.
Упорядоченные множества: упорядочены по параметру score.
Работа с Redis CLI 
Использование Redis CLI для создания, выбора, модификации, удаления и получения базовой информации об объектах.
Примеры операций: установка значения, извлечение значения, проверка типа значения, проверка существования элемента, извлечение всех полей записи, удаление пары ключ-значение.
Управление временем жизни объектов 
Команда TTL возвращает время жизни объекта.
Команда expire устанавливает время жизни объекта.
Время жизни уменьшается с каждой секундой.
Транзакции в Redis 
Основные команды для транзакций: мульти, exec, discard, watch.
Транзакция выполняется только в случае, если другие клиенты не изменили значение переменной.
Пример использования: запись пакета команд, выполнение команд, получение результатов.
Строковые операции 
Установка значения строки с помощью команды `set`.
Добавление строки в конец текущего значения с помощью `append`.
Извлечение нового значения по ключу с помощью `get`.
Получение длины значения по ключу с помощью `strlen`.
Извлечение подстроки из хранимой строки с помощью `getrange`.
Замена подстроки на другую с помощью `setrange`.
Операции с числами 
Установка, инкрементирование и декрементирование чисел.
Прибавление и вычитание чисел с указанием дополнительных параметров.
Работа со списками 
Создание списка с помощью команды `rpush`.
Добавление значений в список: `hello world`, `hello user`, `hello user`.
Вывод элементов списка с помощью `lrange`.
Извлечение первого элемента списка с помощью `lpop`.
Уменьшение длины списка после извлечения элементов.
Работа с сетами 
Создание сета с помощью команды `sadd`.
Добавление элементов в сет: `banana`, `apple`, `strawberry`.
Разделение элементов по цветам: `yellow things`, `red things`.
Проверка наличия элемента в сете с помощью `ismember`.
Удаление элемента из сета с помощью `srem`.
Объединение сетов с помощью `suion`.
Упорядоченные сеты 
Добавление элементов в упорядоченный сет с параметром `score`.
Сортировка элементов по параметру `score`.
Вывод элементов в порядке возрастания или убывания с помощью `zrange`.
Удаление элементов с определёнными параметрами с помощью `zremrange`.
Хэш-таблицы 
Создание хэш-таблицы с помощью команды `hset`.
Добавление полей в хэш-таблицу: `name`, `email`.
Получение ключей и значений хэш-таблицы с помощью `hkeys` и `hgetall`.
Увеличение значения поля с помощью `hincrby`.
Удаление поля из хэш-таблицы с помощью `hdel`.
Pub/Sub 
Подписка на канал с помощью `subscribe`.
Публикация значения в канал с помощью `publish`.
Получение сообщения из канала с помощью `get`.
